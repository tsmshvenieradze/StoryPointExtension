<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Toolbar iframe → host postMessage | The toolbar entry's `execute(actionContext)` callback receives an opaque, untyped object from the ADO host across the iframe postMessage boundary. The shape is undocumented and varies by surface (single-select toolbar, multi-select array, context menu). |
| Toolbar iframe → modal iframe (via `openCustomDialog` configuration) | The `{ workItemId: number }` payload is serialized to JSON across postMessage. Only JSON-safe primitives survive; the modal must NOT trust unstructured fields beyond `workItemId`. |
| Local filesystem → tfx-cli | `scripts/dev-publish.cjs` reads `.env.local` (gitignored) for `TFX_PAT` and passes it as an argv string to the tfx-cli child process. The PAT must NOT cross the iframe boundary, must NOT be bundled, and must NOT be logged. |
| Repo → git remote | `.env.local` and `*.vsix` must NEVER reach the remote. `.gitignore` is the gate; the dev-publish script enforces a `git ls-files .env.local` runtime check as a second line of defense. |
| Manifest scope at install time | Once published with `vso.work_write`, expanding scope post-install forces every existing user to re-consent — a reputation-damaging UX failure. Phase 2 must NOT add new scopes. |
