# Phase 4 — UI Design Contract

> Visual and interaction contract for the write path. Phase 4 EXTENDS Phase 3's UI-SPEC; everything not re-specified here is inherited unchanged. Generated by gsd-ui-researcher; consumed by planner, executor, ui-checker, and ui-auditor.

> **Scope:** real Apply (comment-first → field-write per Phase 0 D-01), overwrite confirm panel (D-03), read-only branch (D-06), saving overlay (D-15), error banners with Retry (D-08, D-09), permission-warn banner (D-07), success indicator (D-10). Out of scope: CMMI live verification (Phase 5), custom-field rename support (PROJECT.md Out of Scope), bundle-size CI gate (Phase 5), Marketplace listing assets (Phase 5).

> **Inheritance contract:** Surface(SurfaceBackground.neutral) + Page + Header(TitleSize.Medium) chrome, modal title `Calculate Story Points`, context line shape, `Cancel`/`Apply` ButtonGroup pattern, MessageCard severity → role mapping, theme inheritance, no `color: #...` literals, no deep barrel imports, 8-point spacing scale, custom font sizes 12/13/28, weights 400/600 — ALL inherited from Phase 3 UI-SPEC §"Inherited" and ARE NOT re-stated below.

## View-State Machine (NEW in Phase 4)

The orchestrator (`src/ui/CalcModal.tsx`) drives a single state machine. Each mode is a distinct render branch off the same Surface/Page/Header chrome. The Header + context line + banner stack are rendered for ALL modes EXCEPT `noField` (Phase 3) and `readonly` (Phase 4 D-06), which replace the body wholesale.

```
Mode union (TypeScript):
  type ModalMode =
    | "loading"     // read-path effect in flight — Phase 3 D-24, banners+dropdowns disabled
    | "calculator"  // dropdowns + calc panel + Apply/Cancel — Phase 3 default
    | "confirm"     // ConfirmOverwritePanel replaces calc body — D-03
    | "saving"      // saving overlay covers body, Apply button shows "Saving…" — D-15
    | "saved"       // 200ms success indicator replaces ButtonGroup — D-10
    | "readonly"    // ReadOnlyMessage replaces body — D-06
    | "noField"     // Phase 3 NoFieldMessage replaces body — Phase 3 D-19
    | "commentFail" // banner replaces banner area + Retry on comment leg — D-08
    | "fieldFail"   // banner replaces banner area + Retry on field leg only — D-09
```

### Transition table

| From | Trigger | To | Notes |
|------|---------|-----|-------|
| (initial mount) | SDK ready, read-path effect starts | `loading` | Phase 3 D-24 spinner shows in Header |
| `loading` | resolver returns null both fields | `noField` | Phase 3 D-19 |
| `loading` | `isReadOnly() === true` | `readonly` | D-06 — ReadOnlyMessage replaces body; context line still rendered |
| `loading` | resolver ok + isReadOnly false (or probe failed → permission-warn) | `calculator` | Phase 3 default |
| `calculator` | Apply click + `currentSp == null` | `saving` | D-04 trigger threshold; skip confirm |
| `calculator` | Apply click + `currentSp != null` | `confirm` | D-03 — ConfirmOverwritePanel replaces calc body, dropdowns UNMOUNTED |
| `confirm` | Back click | `calculator` | D-03 — selections preserved (state held in CalcModal) |
| `confirm` | Confirm Apply click | `saving` | D-15 saving sequence begins |
| `saving` | comment POST rejects | `commentFail` | D-08 — banner replaces banner area; calc body or confirm body restored to last-edited state |
| `saving` | comment POST ok + field write rejects | `fieldFail` | D-09 — comment is in audit log; banner shows Retry-field-only |
| `saving` | both writes succeed | `saved` | D-10 — 200ms inline indicator |
| `saved` | 200ms timer elapses | (close dialog via `SDK.notifyDialogResult({ closed: true, sp: N })` then fall back to Esc-hint if host does not honor) | D-10 — programmatic close mechanism verified on cezari |
| `commentFail` | Retry click | `saving` | re-run comment POST with same payload (D-08); fresh `createdDate` from server |
| `commentFail` | Cancel click | (host close affordance) | Phase 3 D-08 / Override 1 — Cancel cannot programmatically dismiss |
| `fieldFail` | Retry click | `saving` (field-only) | re-run `setFieldValue` + `.save()` only (D-09); comment NOT re-posted |
| `fieldFail` | Cancel click | (host close affordance) | comment audit remains; pre-fill works on next reopen (D-16) |

**Mounting rules:**
- In `confirm` mode: dropdowns + calc panel components are **unmounted**, not just hidden (D-03 — keeps body height stable for D-15 overlay logic). Selection state lives in CalcModal, not in the dropdown components, so unmount/remount is lossless.
- In `readonly` mode: dropdowns + calc panel + Apply/Cancel ButtonGroup + banner stack are all hidden. Only Header + context line + ReadOnlyMessage render.
- In `saving` mode: the SavingOverlay is positioned absolutely over the body region (excluding Header + context line). Pointer-events disabled inside the overlay region. The Apply button (or Confirm Apply in `confirm`-then-`saving`) swaps label to `Saving…` and shows inline Spinner.

## Layout (NEW Phase 4 panels)

### Confirm overwrite panel (D-03)

Replaces the calculator body (between context line and ButtonGroup) when `mode === "confirm"`:

```
+-----------------------------------------------------+
| Header — title "Calculate Story Points"             |  <- inherited
+-----------------------------------------------------+
| Context line — work item meta                       |  <- inherited (Current SP still shows ORIGINAL value)
+-----------------------------------------------------+
| (banner stack — only the resolver-fail / read-error |
|  / permission-warn banners; pre-fill banner is      |
|  HIDDEN in confirm mode to reduce noise)            |
+-----------------------------------------------------+
|                                                     |  <- 32px top padding
|     Confirm overwrite                               |  <- 16px section heading (semibold)
|                                                     |  <- 16px gap below heading
|     Current Story Points:    3                      |  <- 13px label + 13px value, 8px between rows
|     New Story Points:        5                      |
|                                                     |  <- 24px gap above ButtonGroup
+-----------------------------------------------------+
| ButtonGroup (right-aligned): [Back] [Confirm Apply] |  <- 24px gap above buttons (inherited rule)
+-----------------------------------------------------+
```

**Specifics:**
- Section heading "Confirm overwrite": 16px / weight 600 / line-height 1.2.
- "Current Story Points:" / "New Story Points:" labels: 13px / weight 400 / opacity 0.7 (de-emphasized — matches Phase 3 context-line precedent).
- Numeric values: 13px / weight 600 (semibold, emphasized to draw attention to the diff).
- Two-column row layout: `display: grid; grid-template-columns: max-content 1fr; column-gap: 16px; row-gap: 8px;` so the two values column-align cleanly.
- Container: same `min-width` as the calculator body — no width shrink/grow when the user transitions between calculator and confirm (prevents host dialog re-resize). Achieved by inheriting the parent body container's natural width.
- "Back" button: default style (NOT primary). Left-of-pair.
- "Confirm Apply" button: `primary={true}`. Right-of-pair (mirrors Phase 3's Cancel/Apply order — primary on the right per Microsoft Fluent guidelines).
- Apply order is reversed semantically from Phase 3's Cancel/Apply: Back is the "abort destructive intent" button (returns to calculator with selections preserved). Confirm Apply triggers the write.

### Read-only branch (D-06)

When `mode === "readonly"`: the entire calculator UI (3 dropdowns + calc panel + Apply/Cancel ButtonGroup + banner stack) is hidden. Replaced by a centered MessageCard, parallel shape to Phase 3's `NoFieldMessage.tsx`:

```
+-----------------------------------------------------+
| Header — title "Calculate Story Points"             |  <- inherited
+-----------------------------------------------------+
| Context line — work item meta                       |  <- inherited (Current SP still visible — read-only inspection)
+-----------------------------------------------------+
|                                                     |  <- min-height 280px container (matches NoFieldMessage)
|     [ⓘ MessageCard Info — read-only message]       |  <- centered horizontally, vertically
|                                                     |
|     Press Esc or click outside to close.            |  <- 11px / opacity 0.5 / 8px below MessageCard
|                                                     |
+-----------------------------------------------------+
```

- Container: `display: flex; flex-direction: column; align-items: center; justify-content: center; min-height: 280px;` (parallel to Phase 3 NoFieldMessage).
- No Close button. The read-only branch does not include an action button because there is no calculation to perform; the host's Esc / X / outside-click are the close affordances. Hint text below the MessageCard makes this discoverable, mirroring Phase 3's no-field hint pattern (Phase 3 §"No-field state").
- Banner stack is HIDDEN in this mode — the user has nothing to act on; banners would be noise.

### Saving overlay (D-15)

Renders absolutely positioned over the body region (excluding Header + context line) whenever `mode === "saving"`:

```
+-----------------------------------------------------+
| Header — title "Calculate Story Points"             |  <- visible, NOT covered
+-----------------------------------------------------+
| Context line — work item meta                       |  <- visible, NOT covered
+-----------------------------------------------------+
| ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ |  <- ~30% opacity neutral overlay starts here
| ░ (banner stack — visible underneath, dimmed)     ░ |
| ░ (dropdowns + calc panel OR confirm panel —      ░ |
| ░  visible underneath, dimmed; pointer-events:    ░ |
| ░  none; aria-hidden="true")                      ░ |
| ░                                                  ░ |
| ░          [Spinner medium]                        ░ |  <- centered Spinner over the dimmed body
| ░          Saving…                                 ░ |  <- 13px below spinner, weight 400, opacity 0.8
| ░                                                  ░ |
| ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ |
+-----------------------------------------------------+
| ButtonGroup: [Cancel disabled] [Saving… spinner]    |  <- buttons remain rendered, swapped to disabled state; Apply text becomes "Saving…" with inline Spinner glyph
+-----------------------------------------------------+
```

**Overlay implementation:**
- Backdrop: `position: absolute; inset: 0; background: var(--surface-background-color); opacity: 0.6;` — uses the host theme variable so the dim adapts to light/dark. (Note: the result is ~60% surface over the body, giving the desired ~30-40% perceived dim — a literal `opacity: 0.3` on the backdrop produces too little contrast in dark theme; the inverse-tinted dim is the standard pattern for theme-aware overlays.)
- Pointer events: `pointer-events: none` on the underlying body container, OR `pointer-events: auto` on the overlay div with no children that accept clicks. Both achieve the same "no interaction" goal; planner picks based on whether the overlay needs to block click events at the React level vs CSS level.
- ARIA: the overlay div carries `role="status"` and `aria-live="polite"`; the underlying body region gets `aria-busy="true"` while saving.
- Spinner: `<Spinner size={SpinnerSize.medium} ariaLabel="Saving" />`. Centered horizontally and vertically inside the overlay.
- "Saving…" text below spinner: 13px / weight 400 / opacity 0.8 / single Unicode ellipsis (U+2026, NOT three ASCII dots). Matches Phase 3 dropdown placeholder ellipsis convention.

**Apply / Cancel button states during saving:**
- Apply button (or Confirm Apply): label swapped to `Saving…` (Unicode ellipsis), `disabled={true}`, an inline `<Spinner size={SpinnerSize.small}>` rendered before the label. Primary styling preserved (still ADO blue) so the button is recognizable as the in-flight action.
- Cancel / Back button: `disabled={true}`. No label change.

**Host close affordances during saving (D-15):**
- Planner attempts to set `lightDismiss: false` in the original `openCustomDialog` options before opening (toggling at runtime is unsupported per Phase 3 Override 1).
- If the host honors `lightDismiss: false`: Esc / X / outside-click are blocked while the dialog is rendered.
- If not (likely fallback): the dialog can still be force-closed by the user. Mid-write force-close orphans the in-flight fetch; the request still completes server-side. This is an acknowledged Phase 4 limitation per CONTEXT D-15 — recoverable because (a) the saving window is sub-second on the happy path, (b) extra comments are benign per Phase 0 D-03, (c) the field write is idempotent.

### Success indicator (D-10)

When both writes succeed, the `ButtonGroup` is replaced (not overlaid) by an inline indicator for 200ms:

```
+-----------------------------------------------------+
| (calculator body remains visible — no transition    |
|  flash; just the bottom action row swaps)           |
+-----------------------------------------------------+
| (right-aligned) Saved ✓                             |  <- replaces ButtonGroup, no layout shift
+-----------------------------------------------------+
```

- Text: `Saved ✓` (U+2713 check-mark, NOT ASCII `v` or `+`).
- Style: 14px / weight 600 / `color` inherits `var(--status-success-foreground)` from the host theme (semantic green). NOT a custom hex color.
- Position: same right-aligned slot as the ButtonGroup. The two render in the same flex container; only one mounts at a time. This guarantees zero layout shift in the 200ms window (per CONTEXT discretion: "Aim for minimal layout shift in the 200ms window").
- Duration: exactly 200ms via `setTimeout`. After elapse, call `SDK.notifyDialogResult({ closed: true, sp: N })` (or whatever shape cezari verification confirms the host accepts as "close-the-dialog"). If the host does not dismiss, fall back to a hint line `Press Esc to close.` rendered in the same slot for the remainder of the dialog lifetime — cezari verification (D-17 scenario 1) determines which branch applies.
- Banner stack remains rendered during the 200ms — the user briefly sees both the success indicator AND any informational banners (pre-fill from the just-written sentinel will not show because the modal is about to close).

### Error banners (D-08, D-09)

Error banners pin to the top of the body region, replacing the banner-stack space (above dropdowns / confirm panel). They render in addition to — not in place of — any informational banners (resolver-fail, read-error, permission-warn) that were already showing; the error banner takes the topmost position.

**Comment-fail banner (D-08):** Replaces the body layout when `mode === "commentFail"`. Calculator selections (or confirm panel state, if the failure happened post-confirm) remain UNCHANGED. The banner sits above the calculator body / confirm panel.

```tsx
<MessageCard severity={MessageCardSeverity.Error}>
  Could not save audit comment. {friendlyMessage} (HTTP {status})
  <ButtonGroup>
    <Button text="Retry" primary={true} onClick={handleCommentRetry} />
  </ButtonGroup>
</MessageCard>
```

- Banner is NOT dismissable (`onDismiss` omitted) — the user must Retry or Cancel out.
- Retry button: primary styling (the only "do something forward" affordance available). Re-runs the comment POST with the same payload (server timestamps a new `createdDate`).
- The Apply / Cancel ButtonGroup at the bottom is REPLACED by `[Cancel] [Retry]` while in `commentFail` mode (Apply makes no sense — the user must address the failure first). Implementation: the orchestrator swaps the ButtonGroup contents based on mode.
- Cancel button stays default style.

**Field-fail banner (D-09):** Same shape as comment-fail, distinct copy.

```tsx
<MessageCard severity={MessageCardSeverity.Error}>
  Audit comment recorded. The Story Points field could not be updated. {friendlyMessage} ({httpOrSdkLabel})
  <ButtonGroup>
    <Button text="Retry" primary={true} onClick={handleFieldRetry} />
  </ButtonGroup>
</MessageCard>
```

- `httpOrSdkLabel`: either `HTTP {status}` for HTTP-mapped failures, or `SDK error: {className}` for SDK-class-mapped failures (D-20). Mapper in `src/apply/errorMessages.ts` decides based on input shape.
- Retry runs the field write only (`setFieldValue` + `.save()`), NOT the comment POST. Comment is intentionally kept (per CONTEXT D-09 rationale: comment is already in the audit log, becomes pre-fill source on next reopen via D-16).

### Permission-warn banner (D-07)

Banner-stack item between read-error and pre-fill (NEW slot in the inherited stack):

**Updated banner stack ordering (extends Phase 3):**

```
1. resolver-fail (Warning) — Phase 3 D-20
2. read-error (Warning) — Phase 3 D-25
3. permission-warn (Warning) — NEW Phase 4 D-07
4. pre-fill (Info) — Phase 3 D-12
```

```tsx
<MessageCard
  severity={MessageCardSeverity.Warning}
  onDismiss={() => setPermissionWarnDismissed(true)}
>
  Could not verify your permissions — Apply may fail if this work item is read-only.
</MessageCard>
```

- `onDismiss` is defined → MessageCard renders a built-in dismiss X. User can dismiss; no auto-dismiss.
- Renders only when `readResult.permission?.probeFailed === true` (i.e., `isReadOnly()` rejected/threw/timed out — not when it returned `false`).
- Does NOT block the user — `mode` stays `calculator`; Apply still attempts the write. The actual write's error path (D-08, D-09) takes over with a precise message if the optimistic assumption was wrong.

## Color

Phase 4 introduces **zero custom hex colors**. Like Phase 3, every visible color flows from the host ADO theme via `azure-devops-ui` CSS variables.

| Role | Value | Phase 4 usage |
|------|-------|---------------|
| Dominant (60%) | `var(--background-color)` | Modal body background (inherited) |
| Secondary (30%) | `var(--callout-background-color)` | Dropdown callouts; calc panel border (inherited) |
| Accent (10%) | `var(--communication-background)` (theme blue) | **Apply** primary button AND **Confirm Apply** primary button AND **Retry** primary button. Three buttons total may carry primary styling across the entire write-path UI; only one is rendered at any given moment (mode-mutually-exclusive). |
| Destructive | not used | Phase 4 has no destructive-typed actions. Confirm Apply is "intent-confirmation," not "destructive-confirmation" — the action is reversible (the user can recalculate and re-apply). No red Button styling. |
| Severity Info | `var(--status-info-foreground)` | Read-only message MessageCard |
| Severity Warning | `var(--status-warning-foreground)` | Permission-warn banner |
| Severity Error | `var(--status-error-foreground)` | Comment-fail banner; Field-fail banner |
| Severity Success | `var(--status-success-foreground)` | "Saved ✓" check-mark glyph color |

**Accent reserved for** — the SPECIFIC primary buttons across the write path:
1. **Apply** button (Phase 3 `calculator` mode)
2. **Confirm Apply** button (Phase 4 `confirm` mode)
3. **Retry** button (Phase 4 `commentFail` and `fieldFail` modes)

No other element carries `primary={true}`. Cancel, Back, Close all use default Button. The Final SP hero uses default text color (as in Phase 3); the success-indicator check-mark uses semantic success-foreground (theme green), which is its own role and does NOT count as accent.

## Loading & Saving States (NEW Phase 4)

The loading state (read-path effect) is inherited from Phase 3 §"Loading & State Sequence". Phase 4 adds the saving state:

```
t=0     User clicks Apply (with currentSp != null)
        → mode transitions calculator → confirm
        → ConfirmOverwritePanel mounts, calc body unmounts
        → focus moves to Confirm Apply button

t=0+    User clicks Confirm Apply
        → mode transitions confirm → saving
        → SavingOverlay mounts over body
        → Apply button label swaps to "Saving…", disabled, inline Spinner
        → Cancel/Back disabled
        → aria-busy=true on body
        → console: [sp-calc/apply] postComment start ...
        → adoFetch POST /comments?api-version=7.0-preview.3 begins

t=t1    POST resolves
        → on 2xx: console: [sp-calc/apply] postComment ok commentId=...
                   begin field-write leg
                   console: [sp-calc/apply] setFieldValue start ...
        → on non-2xx: mode transitions saving → commentFail
                       SavingOverlay unmounts
                       CommentFailBanner mounts above body
                       Retry button focused
                       STOP (no field write attempted per D-08)

t=t2    setFieldValue + .save() resolves
        → on success: console: [sp-calc/apply] both writes succeeded
                       mode transitions saving → saved
                       SavingOverlay unmounts
                       SavedIndicator mounts in ButtonGroup slot
                       200ms timer starts
        → on rejection: console: [sp-calc/apply] setFieldValue failed sdkClass=... message=...
                         mode transitions saving → fieldFail
                         SavingOverlay unmounts
                         FieldFailBanner mounts above body
                         Retry button focused (this Retry runs field-only)
                         STOP

t=t2+200ms  SavedIndicator timer elapses
        → SDK.notifyDialogResult({ closed: true, sp: N }) attempted
        → if dialog dismisses: work item form re-renders SP via SDK.setFieldValue boundary
        → if dialog persists: SavedIndicator replaced with "Press Esc to close." hint
```

**Read-path → readonly transition (NEW Phase 4):**

```
t=t1.5  isReadOnly() probe resolves (parallel to other read probes)
        → on true: mode transitions loading → readonly directly
                    (skips calculator mount entirely)
                    ReadOnlyMessage renders
        → on false: mode continues to calculator (Phase 3 default)
        → on rejection/timeout: readResult.permission.probeFailed = true
                                 mode continues to calculator
                                 PermissionWarnBanner shows in stack
```

**Read-only check ordering** in CalcModal render path:
```
if (readResult.fieldRefName === null) → noField (Phase 3)
else if (readResult.permission?.isReadOnly === true) → readonly (Phase 4 D-06)
else → calculator (Phase 3 default)
```

The no-field check takes precedence so that "no SP field on this type" wins over "user can't write" — the message is more accurate.
