  <how-to-verify>
    **Setup:**
    1. Open `https://cezari.visualstudio.com/Cezari` in Chrome (or Edge — both Chromium-based; matches what most Marketplace users have).
    2. Confirm the new extension version shows under your user Profile → Manage Extensions for the `cezari` org. If it doesn't, Marketplace validation may still be running — wait 60 seconds and refresh the work item page once.
    3. Open the browser DevTools Console BEFORE clicking the toolbar button — you'll need to see the `[sp-calc/...]` log output for several checks.

    **Run the 12-item checklist (D-29). Mark each PASS or FAIL with evidence (screenshot or console log paste) in `03-VERIFICATION.md`:**

    1. **Modal opens on click for User Story, Bug, Task, Feature, Epic.**
       - For each of the 5 work item types: open one work item of that type; confirm "Calculate Story Points" entry in the toolbar menu (the `…` menu); click it; confirm the modal opens.
       - Repeat for at least one work item per type. If `cezari` doesn't have one of the types, create one (or skip with a note — these are standard Agile types and should exist).

    2. **Three dropdowns render with `Select level…` placeholders, populated with the five LEVELS in Very Easy → Very Hard order.**
       - In the just-opened modal, confirm three labeled rows: Complexity, Uncertainty, Effort.
       - Each shows `Select level…` (note: U+2026 ellipsis, single character — it should look like a single dot-dot-dot, not three separate dots).
       - Click any dropdown; confirm the five options in order: Very Easy, Easy, Medium, Hard, Very Hard.

    3. **Calc panel renders with em-dash placeholders + formula text from open.**
       - Before any selection, the panel shows: "Final Story Points" → `—`, "W = —", "Raw SP = —".
       - Both formula lines are visible: `W = 0.4·C + 0.4·U + 0.2·E` and `SP = round_fib(0.5 × 26^((W−1)/4))`.
       - The middle-dot, times, and minus characters render correctly (look like proper Unicode glyphs, not ASCII fallbacks).

    4. **Selecting all three updates W, Raw SP, Final SP live (matches `calculate()` output to 2 decimals on the intermediates).**
       - Select Complexity=Hard, Uncertainty=Medium, Effort=Easy.
       - Sanity-check: with C=4, U=3, E=2 → W = 0.4·4 + 0.4·3 + 0.2·2 = 1.6 + 1.2 + 0.4 = 3.20.
       - Raw SP = 0.5 · 26^((3.2−1)/4) = 0.5 · 26^0.55 ≈ 0.5 · 5.78 ≈ 2.89.
       - Final SP per the threshold table (≤4→3) = 3.
       - Confirm the panel reads: W = 3.20, Raw SP = 2.89, Final Story Points = 3.

    5. **With a prior sentinel comment present, dropdowns auto-fill and the provenance banner shows the absolute date.**
       - Manually post a comment on the test work item containing this exact text (paste into the Discussion field):
         ```
         <!-- sp-calc:v1 {"sp":5,"c":"Very Hard","u":"Hard","e":"Easy","schemaVersion":1} -->
         Story Points: 5 (Complexity=Very Hard, Uncertainty=Hard, Effort=Easy)
         ```
       - Save the comment.
       - Close the modal if open, then re-open via the toolbar button.
       - Confirm the three dropdowns populate as: Complexity=Very Hard, Uncertainty=Hard, Effort=Easy.
       - Confirm the Info banner reads: `Pre-filled from your last calculation on {today's date as "Month D, YYYY"}.` — e.g., `Pre-filled from your last calculation on May 2, 2026.`
       - Confirm the calc panel reads W = 3.40, Raw SP = 3.32, Final Story Points = 3 (the panel shows the LIVE calc, not necessarily the sentinel's `sp:5` — that's only the prior-write history; live output reflects the current LEVELS in the dropdowns).

    6. **With a manually-edited SP field that diverges from the prior sentinel, banner adds the mismatch note.**
       - On the same work item with the sentinel comment from step 5 (sentinel sp=5), set the Story Points field directly via the work item form to a different value, e.g., 8. Save.
       - Re-open the modal.
       - Confirm the banner now reads: `Pre-filled from your last calculation on May 2, 2026. Field currently shows 8 — may have been edited directly.` (em-dash U+2014, not ASCII).
       - Severity stays Info (not Warning) — same blue/info icon as step 5.

    7. **Apply is disabled until 3rd dropdown selected.**
       - Open a fresh work item with no sentinel comment.
       - Initially the Apply button is grayed out (disabled).
       - Pick Complexity=Medium → Apply still disabled.
       - Pick Uncertainty=Easy → Apply still disabled.
       - Pick Effort=Hard → Apply becomes enabled (primary blue).

    8. **Stub-Apply console-logs the right payload.**
       - With all three selected, click Apply.
       - Confirm DevTools Console shows: `[sp-calc/apply] would write SP=2, fieldRefName=Microsoft.VSTS.Scheduling.StoryPoints, comment=<!-- sp-calc:v1 {"sp":2,"c":"Medium","u":"Easy","e":"Hard","schemaVersion":1} -->\nStory Points: 2 (Complexity=Medium, Uncertainty=Easy, Effort=Hard)`
       - Note: D-27 dialog-close is not implemented (no programmatic close API). The dialog stays open after Apply; user closes via X / Esc.

    9. **Cancel closes immediately, no confirm.**
       - Open a fresh modal; pick some selections; click Cancel.
       - Confirm: no "discard?" prompt appears. Console shows: `[sp-calc/modal] cancel clicked — host close affordance required`.
       - Per Override 1, Cancel cannot programmatically close — user uses X / Esc / outside-click. Confirm Esc closes the dialog cleanly.

    10. **Tab/Enter/Esc keyboard nav works (UI-07).**
       - Open a fresh modal. After load completes, focus should be on the first dropdown (Complexity).
       - Press Tab → focus moves to Uncertainty dropdown.
       - Press Tab → focus moves to Effort dropdown.
       - Press Enter on a focused dropdown → callout opens.
       - Press Up/Down arrows in the callout → highlight moves.
       - Press Enter → option is selected, callout closes.
       - Press Esc with the callout open → callout dismisses, focus stays on trigger (does NOT close dialog).
       - Press Esc with focus on the trigger (callout closed) → host dialog closes.
       - Tab through to Apply → Enter triggers stubApply (console log appears).

    11. **Modal renders with no scrollbar at the configured size (D-06).**
       - In a fresh modal with no banners (no prior sentinel), confirm no vertical scrollbar in the dialog body.
       - In the pre-filled + mismatch-banner case (step 6 state), confirm STILL no vertical scrollbar (this is the worst-case height — banner + 3 dropdowns + calc panel + button row).
       - If a scrollbar appears in the worst-case state, document it in 03-VERIFICATION.md as a FAIL on D-06; the team may need to adjust `minHeight: 380` in `CalcModal.tsx` or compress vertical spacing. (Phase 3 doesn't have to fix this in-plan; document and assess.)

    12. **Theme toggle (light↔dark) flips modal colors as in Phase 2.**
       - Open the modal in light theme → note colors. Switch ADO host theme to dark via user menu → User Settings. Reload the work item page. Re-open the modal.
       - Confirm: background flips dark, text flips light, dropdown trigger borders adjust, MessageCard banner backgrounds tint appropriately, Apply primary button blue stays correct (slightly adjusted for dark contrast).
       - Run the same check across all 6 new components per UI-SPEC §Theme Inheritance Verification table (Dropdown closed/open, MessageCard Info/Warning, Spinner, Button primary/default, ButtonGroup, FormItem label, Calc panel border).
       - If any component renders as a "white box on dark theme" or "plain unstyled text," that's the deep-import escape hatch (Pitfall 4) — fix by switching to barrel imports in the offending file. Phase 3 SHOULD have caught this in Plan 3's automated checks but theme issues only manifest on the live host.
