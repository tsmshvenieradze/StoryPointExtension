# Phase 3 — UI Design Contract

> Visual and interaction contract for the Story Point Calculator modal. Generated by gsd-ui-researcher; consumed by planner, executor, ui-checker, and ui-auditor.

> **Stack note:** This is a React 18 Azure DevOps extension. The "design system" is `azure-devops-ui@2.272.0`, the official Microsoft component library; shadcn does NOT apply. All color and most typography tokens come from the host ADO iframe via theme CSS variables (verified Phase 2: Surface + Page + Header inherit theme automatically). Custom CSS is forbidden where an `azure-devops-ui` primitive exists. Spacing follows an 8-point scale on top of the library's chrome.

## Layout (D-01 — locked)

Stacked vertical, single column, top-to-bottom inside `Surface > Page`:

```
+-----------------------------------------------------+
| Header — title "Calculate Story Points" (D-02)      |  <- TitleSize.Medium
+-----------------------------------------------------+
| Context line — work item meta (D-03)                |  <- 13px / weight 400, opacity 0.7
+-----------------------------------------------------+
| MessageCard banner (conditional: pre-fill OR        |  <- 16px gap below context line; only when present
|   read-error OR resolver-fail)                      |
+-----------------------------------------------------+
| FormItem: Complexity   [Dropdown]                   |  <- 16px gap above first dropdown
| FormItem: Uncertainty  [Dropdown]                   |  <- 16px gap between dropdowns
| FormItem: Effort       [Dropdown]                   |
+-----------------------------------------------------+
| Calculation Details panel (D-04, D-05)              |  <- 24px gap above panel; bordered region
|   Hero: Final SP                                    |
|   Sub: W = X.XX   Raw SP = X.XX                     |
|   Formula: W = 0.4·C + 0.4·U + 0.2·E …              |
+-----------------------------------------------------+
| ButtonGroup (right-aligned): [Cancel] [Apply]       |  <- 24px gap above buttons; Override 1
+-----------------------------------------------------+
```

**Width / height contract (D-06):**

- Width: inherit host dialog default (Phase 2 "small" footprint). Do NOT pass explicit width.
- Height: `IDialogOptions` exposes no height knob (Override 1). Set the modal body's `min-height` to **380px** so the layout above plus the third dropdown's callout fits without clip; rely on the host dialog's content-fit behavior.
- **Hard rule:** the modal MUST NOT show a vertical scrollbar at any state (calc panel filled, banner visible, all three callouts closed). Verifier checks this on cezari.
- Dropdown callout overflow: planner verifies on cezari that the third dropdown's callout auto-flips upward when there is no room below (Pitfall 2). If it does not, set `dropdownOrigin={{ horizontal: 0, vertical: 1 }}` on dropdown 3 to force upward flip.

## Spacing Scale (8-point)

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps inside the calc panel (between number and unit) |
| sm | 8px | Gap between Header and context line; gap inside MessageCard around the icon |
| md | 16px | Gap between FormItems (dropdown rows); gap between context line and banner; gap between banner and first dropdown; horizontal padding inside calc panel |
| lg | 24px | Gap above Calculation Details panel; gap above ButtonGroup; vertical padding inside calc panel |
| xl | 32px | Reserved — not used in Phase 3 |

Exceptions: none for Phase 3. The `Page` component already supplies its own outer padding via `page-content page-content-top` classes (Phase 2 pattern); do NOT add additional outer padding.

Spacing is implemented via inline `style={{ marginTop: 16 }}` on container `<div>` wrappers — NOT via custom CSS files. No new `.css` / `.scss` files are introduced in Phase 3 except theme-token-aware inline styles as documented below.

## Color

Phase 3 introduces **zero custom colors**. Every visible color flows from the host ADO theme via `azure-devops-ui`'s CSS variables. The 60/30/10 split below is observational — it describes how the library's tokens render, not new tokens we declare.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background-color)` (host theme) — applied by `Surface` | Modal body background |
| Secondary (30%) | `var(--callout-background-color)` (host theme) — applied by Dropdown callout, Calc panel border | Dropdown callouts; calc panel subtle bordered region |
| Accent (10%) | `var(--communication-background)` (host theme blue) — applied by `Button primary={true}` | **Reserved exclusively for the Apply button.** No other element in the modal uses this color. |
| Destructive | not used in Phase 3 | Phase 4 (overwrite confirm + write errors) introduces destructive treatment |
| Severity Info | `var(--status-info-foreground)` — applied by `MessageCard severity={MessageCardSeverity.Info}` | Pre-fill banner; no-field message |
| Severity Warning | `var(--status-warning-foreground)` — applied by `MessageCard severity={MessageCardSeverity.Warning}` | Read-error banner; FieldResolver-fail message |

**Accent reserved for:** the **Apply** primary button only. Cancel uses default Button (no accent). Final SP hero uses default text color, NOT accent — its emphasis comes from size + weight, not color.

**Severity-to-MessageCard mapping (locked):**

| Banner | Severity | Trigger |
|--------|----------|---------|
| Pre-fill provenance (D-12, D-14) | `MessageCardSeverity.Info` | `parseLatest(comments)` returned a valid payload |
| Read-error (D-25) | `MessageCardSeverity.Warning` | `getCommentsModern()` rejected |
| No-field (D-19, Override 3) | `MessageCardSeverity.Info` | `FieldResolver` returned `null` for both standard fields |
| FieldResolver-fail (D-20) | `MessageCardSeverity.Warning` | `getFields()` rejected — defaulted to StoryPoints |

> Mismatch addendum (D-14) does NOT change severity — stays Info on the same pre-fill banner.

**Rules:**
- No `color: #...` literals anywhere in Phase 3 source.
- No `background-color:` declarations except via inheriting the library's component CSS.
- `opacity` is allowed (used at 0.7 on de-emphasized text per Phase 2 precedent).

## Component Contracts

### Pre-fill banner (Info MessageCard)

```tsx
<MessageCard
  severity={MessageCardSeverity.Info}
  onDismiss={() => setBannerDismissed(true)}
>
  {`Pre-filled from your last calculation on ${formattedDate}.`}
  {currentSpMismatch !== null
    ? ` Field currently shows ${currentSpMismatch} — may have been edited directly.`
    : null}
</MessageCard>
```

- `onDismiss` defined → MessageCard renders a built-in dismiss X.
- Banner does NOT auto-dismiss.
- Render only when `parseLatest()` returned non-null AND payload `c/u/e` are valid `Level`s.

### Read-error banner (Warning MessageCard)

```tsx
<MessageCard severity={MessageCardSeverity.Warning}>
  Could not load prior calculations — starting fresh.
</MessageCard>
```

- No dismiss prop → not dismissable; persists for the dialog lifetime.
- Renders mutually-exclusively with the pre-fill banner (no comments → no pre-fill possible anyway).

### FieldResolver-fail banner (Warning MessageCard)

```tsx
<MessageCard severity={MessageCardSeverity.Warning}>
  Could not detect field type — assuming Microsoft.VSTS.Scheduling.StoryPoints.
</MessageCard>
```

- Renders **above** any pre-fill or read-error banner (stack order: resolver-fail → read-error → pre-fill, top to bottom).

### No-field state (Info MessageCard + Close button)

When `FieldResolver` returns `null` for both `Microsoft.VSTS.Scheduling.StoryPoints` AND `Microsoft.VSTS.Scheduling.Size`:

- The entire calculator UI (3 dropdowns + calc panel + Apply/Cancel ButtonGroup) is **replaced** by a centered Info MessageCard with the locked no-field copy and one default-style `Close` Button below it.
- Vertical centering: wrap MessageCard + Close button in a flex container with `display: flex; flex-direction: column; align-items: center; justify-content: center; min-height: 280px;`.
- Close button reload: since `IHostPageLayoutService` has no `closeCustomDialog`, Close cannot programmatically dismiss the host dialog. The Close button instead renders a hint below itself: small text `Press Esc or click outside the dialog to close.` (matches Phase 2 modal hint pattern). This is a known limitation of Override 1 and is acceptable for the no-field path because the user has no calculation work to perform.

### Three Dropdowns (FormItem-wrapped)

```tsx
<FormItem label="Complexity">
  <Dropdown<{ level: Level }>
    items={LEVELS.map(level => ({ id: level, text: level, data: { level } }))}
    placeholder="Select level…"
    onSelect={(_event, item) => setComplexity(item.data!.level)}
    selection={complexitySelection}
    disabled={isLoading}
    ariaLabel="Complexity level"
  />
</FormItem>
```

- `LEVELS` iterated in array order = Very Easy → Very Hard (D-10, no reverse).
- During the read path (`isLoading` true), all three Dropdowns are `disabled` (D-24). A `Spinner` of `SpinnerSize.medium` renders in the Header row (right-aligned next to the title) while loading. No skeleton shapes — `azure-devops-ui` has no skeleton primitive; the disabled state plus the spinner is the loading affordance.
- `ariaLabel` declared on each Dropdown for screen-reader independence from the FormItem label. Use `Complexity level`, `Uncertainty level`, `Effort level`.

### Calculation Details panel (D-04, D-05)

Layout inside a single `<section>` with theme-aware bordered styling:

```
[ FINAL STORY POINTS ]      <- 12px uppercase label, opacity 0.7
[       3       ]            <- 28px hero, semibold, default color
W = 1.85    Raw SP = 0.83    <- 13px sub-row, two columns
W = 0.4·C + 0.4·U + 0.2·E    <- 12px formula, opacity 0.7, always visible
SP = round_fib(0.5 × 26^((W−1)/4))
```

- Border: `border: 1px solid var(--callout-border-color); border-radius: 4px;` — uses host theme variable, so it adapts to light/dark.
- Padding: 16px horizontal, 24px vertical.
- Internal vertical rhythm: 4px between hero label and hero number, 8px between hero number and sub-row, 16px between sub-row and formula block.
- W and Raw SP both formatted to 2 decimals via `n.toFixed(2)` per UI-04. Final SP rendered as integer when ≥ 1, as `0.5` when half-Fibonacci.
- Empty state (any of c/u/e undefined): hero number is `—`, sub-row is `W = —    Raw SP = —`. Formula stays visible.
- Panel always renders with the same height regardless of fill state — no layout shift.

### ButtonGroup (Override 1, body-rendered footer)

```tsx
<div style={{ marginTop: 24, display: "flex", justifyContent: "flex-end" }}>
  <ButtonGroup>
    <Button text="Cancel" onClick={handleCancel} ariaLabel="Cancel and close dialog" />
    <Button text="Apply" primary={true} onClick={handleApplyStub}
            disabled={!isAllSelected} ariaLabel="Apply Story Points to work item" />
  </ButtonGroup>
</div>
```

- Right-aligned per ADO convention (Marketplace dialog samples).
- Apply order: Cancel left, Apply right (primary on the right per Microsoft Fluent guidelines).
- Apply `disabled={complexity === undefined || uncertainty === undefined || effort === undefined}`.
- Cancel cannot programmatically close the host dialog (Override 1 / RESEARCH.md Pitfall 1). On Cancel click, the handler logs `[sp-calc/modal] cancel clicked — host close affordance required` and renders a small hint below the ButtonGroup: `Press Esc or click outside to close.` This hint is permanent (always visible, doesn't toggle) at 11px opacity 0.5, right-aligned beneath the buttons. **Rationale:** the host's X / Esc / `lightDismiss` is the actual close mechanism; the body Cancel button is UX redundancy that satisfies UI-05 literally but cannot mutate dialog state.

### Spinner during read path (D-24)

```tsx
{isLoading && (
  <Spinner size={SpinnerSize.medium} ariaLabel="Loading prior calculation" />
)}
```

- Position: inside the Header row, after the title text, right-aligned.
- Lifetime: from `SDK.ready()` resolved through both `getFields()` and `getCommentsModern()` settled (success or failure). Banner area stays empty during loading (no spinner there to avoid double-spinner clutter).

## Keyboard Navigation Contract (UI-07)

`azure-devops-ui` Dropdown is built on FocusZone + ListBox with native keyboard handling (verified in `Dropdown.d.ts`). The contract below documents EXPECTED behavior; the planner verifies on cezari that each item works without custom keydown handlers.

| Key | Focus location | Expected behavior |
|-----|----------------|-------------------|
| Tab (forward) | anywhere | Moves focus to next focusable: Dropdown 1 → Dropdown 2 → Dropdown 3 → Cancel → Apply → (host chrome). MessageCard dismiss X is also focusable and falls between context line and Dropdown 1 if the banner is rendered. |
| Tab (backward) | anywhere | Reverse of above |
| Enter | Dropdown trigger (closed) | Opens callout |
| Up/Down | Inside callout | Moves highlight |
| Enter | Inside callout | Confirms highlighted option, closes callout, focus returns to trigger |
| Esc | Inside callout | Dismisses callout, focus returns to trigger; does NOT close dialog |
| Esc | On trigger or anywhere outside callout | Closes the host dialog (host-level behavior — works because `lightDismiss: true` from Phase 2) |
| Tab to Apply + Enter | Apply button focused | Triggers stub-Apply (D-27): logs payload, attempts host dialog close (limited per Override 1) |
| Space | Button focused | Same as Enter |

**Custom keydown handlers:** none. The `azure-devops-ui` Dropdown's native handling is sufficient. If verification on cezari shows a gap, planner adds a focused fix and documents it as an addendum.

- Do NOT trap focus inside the modal manually — the host dialog handles focus trapping.

## Loading & State Sequence

```
t=0    Modal iframe spawns; SDK.init({loaded:false}) called
t=t1   SDK.ready() resolved
       → render skeleton: Header (with Spinner), context line "Loading…",
         3 disabled Dropdowns (with placeholder), empty calc panel with em-dashes
t=t2   FieldResolver.getFields() probe completes
       → on success: typeName resolved, banner area still loading (Spinner stays)
       → on failure: render FieldResolver-fail Warning banner; assume StoryPoints; continue
       → on null both fields: REPLACE entire layout with no-field message UI; STOP
t=t3   getFieldValue(System.Title) + getFieldValue(SP) both resolved
       → context line updates from "Loading…" to real values
t=t4   getCommentsModern() resolved
       → on success: parseLatest(); if valid payload, populate trio + render pre-fill banner
       → on success but malformed: log warn; modal stays empty; no banner
       → on failure: render read-error Warning banner; modal stays empty
t=t5   isLoading=false; Spinner unmounts; Dropdowns become enabled
       → if pre-fill set all three, Apply enables and autofocuses; otherwise first Dropdown autofocuses
t=tN   notifyLoadSucceeded() called once per dialog open (Phase 2 lifecycle preserved)
```

`t1..t5` are the read-path latency window (D-24). The user sees structure immediately at `t1` and never a blank dialog.

## Override Acknowledgements

**Override 1 (D-08 — Apply/Cancel in host dialog footer):** Verified infeasible. `IDialogOptions` exposes only `title`, `configuration`, `lightDismiss`, `onClose`. `IHostPageLayoutService` has no `closeCustomDialog`. **Resolution:** Apply + Cancel render in the modal body via `azure-devops-ui` `ButtonGroup` (right-aligned, primary Apply on right). The host's X / Esc / lightDismiss is the actual close affordance; body Cancel is UX redundancy with a permanent hint `Press Esc or click outside to close.` below the buttons.
